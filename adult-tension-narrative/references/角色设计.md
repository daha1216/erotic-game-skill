## 完整开局规则

执行新开局时完整运行 1-14。表格是抽样种子，不是合法全集；允许根据已生成身份做半句改写，但不得二次挑选结果。

默认采用“表结构末位自拟”：每个结构表在表内选项之外增加一个“系统自拟”末位。玩家说“强制表内”时取消末位；玩家说“表外全随机”时仍执行全部步骤，但各结构项均自拟。

所有主要角色和参与成熟亲密互动的角色必须明确为 18 岁以上。校园、社团、学园或师生意象必须声明为成年人教育、职业培训或成年社群。

## 开局事务补充规范

本节不改变上述 1-14 的内容，只约束其执行时机。凡上文要求“建立稳定 ID”或“提交初始状态”，均延后到本节的原子落档阶段执行；落档前只保留无正式 ID 的草稿。

### 五阶段流程

| 阶段 | 行为 | 写入正式状态 |
|---|---|---|
| 1. 读取要求 | 固化模式、预锁项、边界与种子 | 否 |
| 2. 生成草稿 | 完成 1-14 及附加内容 | 否 |
| 3. 统一检查 | 检查结构、因果、人物、安全、名字与重复度 | 否 |
| 4. 局部修复 | 仅修错误字段，最多两轮 | 否 |
| 5. 原子落档 | 一次性生成 ID、提交状态并渲染开场 | 是 |

#### 1. 读取要求

```yaml
opening_request:
  mode: default
  locks:
    world_type: null
    setting_shell: null
    npc_identity: null
  player_boundaries: []
  seed: null
```

将玩家明确指定的任意字段加入 `locks`。预锁字段在本次开局中不可改写；若组合困难，只能调整未锁字段。边界同样不可作为修复目标。

#### 2. 生成草稿

依赖顺序固定为：

1. 世界骨架：步骤 1-8。
2. 人物骨架：步骤 9-13。
3. 因果网络：步骤 14、玩家化身、三层压力、开场三勾与当前节点。

所有结果仅写入：

```yaml
opening_draft:
  world: {}
  player: {}
  npcs: []
  relationships: []
  events: []
  current_node: {}
  validation: {}
```

草稿期不得生成正式角色或事件 ID，不得推进时钟或回合，不得写入正式台账。草稿内如需互相引用，使用仅在草稿有效的临时键。

#### 3. 统一检查

按以下层次一次检查完整草稿：

- 结构：1-14 是否完成，字段、人数、功能和关系边是否一致。
- 因果：身份能否解释资源与权限；事件是否来自当前压力；远近事件是否有因果来源。
- 人物：目标、底线、知识、资源、压力反应和行为偏好是否互相兼容。
- 亲密画像：人格、知识、经验、欲望、偏向、表现和自我调节是否独立生成，是否存在刻板绑定。
- 变化：当前状态是否有足够事件证据，是否符合可塑性与变化条件，是否保留回退和不变化的可能。
- 安全：年龄、玩家边界、独立场景许可、安全状态和玩家控制权是否合格。
- 名字：同局不重名，符合文化、地区、年代和社会背景，主配角风格一致且不过度象征性。
- 重复度：名字、读音、身份组合、处境和标志细节不得与近期记录高度重复。

检查结果必须机器可读：

```yaml
validation:
  passed: false
  errors:
    - field: npcs[0].resources
      reason: 身份无法支持最高权限
      repair: 降低权限或为未锁字段补充合法来源
```

每条错误必须定位到具体字段，并给出不触碰预锁项和玩家边界的修复方式。无错误时设置 `passed: true` 和 `errors: []`。

#### 4. 局部修复

每轮只修改 `validation.errors` 指向的字段及其直接依赖，禁止因局部问题重抽整个世界。修复顺序固定为：

1. 年龄、边界和玩家控制权。
2. 世界与身份冲突。
3. 目标、资源和事件冲突。
4. 人物知识越界。
5. 亲密画像绑定错误或变化证据不足。
6. 名字和文化不匹配。
7. 文风和细节重复。

每轮修复后重新执行全部检查，最多两轮。两轮后仍有错误时终止开局，内部记录错误；不得渲染正文或提交任何草稿状态。

#### 5. 原子落档

仅当 `validation.passed: true` 时，一次性完成：生成稳定角色与事件 ID、替换草稿引用、写入世界常量、角色决策卡、关系边、三层压力事件和当前节点，并将完整初始状态提交为回合 0。提交成功后再渲染开场，向玩家显示“回合：1”。

落档是单次事务：任一字段、引用或 ID 生成失败时整体回滚，正式状态保持开局前状态，不得留下部分角色或事件。

## 1-14 开局流程

| 序 | 项目 | 规则 | 产出 |
|---|---|---|---|
| 1 | 世界观档 | 1..3 | 现实 / 半架空 / 强架空 |
| 2 | 主张力引擎 | 张力引擎族与类内项 | 一句可演压力 |
| 3 | 咬合开关 | 1..2 | 单引擎 / 双引擎 |
| 4 | 对冲引擎 | 仅双引擎时生成，族不得与主引擎相同 | 第二条压力 |
| 5 | 制度与场合壳 | 壳类与类内项 | 类型、地点、当前规则与压力 |
| 6 | Tier | 1..3 | 世界追算深度 |
| 7 | 模式 | 1..2 | reliable / immersive |
| 8 | 权力结构 | 1..4 | 玩家高位 / NPC高位 / 对等 / 可切换 |
| 9 | 主 NPC 全维 | 依次执行 9a-9h | 完整角色卡 |
| 10 | 关系阶段与安全字段 | 按当前关系生成 | 关系阶段、许可范围、安全信号 |
| 11 | 配角数 | 1..3 | 数量 |
| 12 | 配角功能 | 每人独立生成 | 功能列表 |
| 13 | 配角建档 | 每人独立建档 | 配角卡 |
| 14 | 关系图 | Tier3 必须生成，否则写明不适用 | 稳定 ID 的边 |

完成 1-14 后继续生成玩家化身、三层压力种子、开场三勾和初始状态。条件不满足的步骤必须显式写 `—` 与原因。

## 张力引擎

先选择一族，再选择族内一项。将结果翻译成“谁拥有什么、谁缺什么、什么即将改变”的一句压力，不要只输出标签。

### 1. 权势结构

1. 名义上级与实际控制者不一致
2. 一方掌握程序许可，另一方掌握现场资源
3. 公开身份高低与私下依赖相反
4. 临时代理权将在期限后失效
5. 多方争夺同一人的支持或证词
6. 权力来源合法，但使用方式会损害声誉

### 2. 名分裂隙

1. 公开关系与真实合作不一致
2. 旧承诺与新身份发生冲突
3. 双方必须维持一段并不完整的对外说法
4. 一方需要另一方承认某种名分，另一方拒绝定义

### 3. 身体与能力通道

1. 一方必须借助另一方才能稳定能力
2. 特殊环境只允许某类人进入
3. 能力使用会留下可追踪痕迹
4. 一方能感知另一方无法隐藏的状态
5. 身体或能力恢复与某个期限绑定

### 4. 资源锁

1. 关键证据只有一份
2. 通行资格必须由特定人物担保
3. 资金、药物、设备或住处存在单点依赖
4. 双方各持一半资源，单独都无法完成目标
5. 资源可以替代，但替代方案会暴露身份

### 5. 时空锁

1. 必须在天亮前完成
2. 每次见面只有固定短窗口
3. 地点即将封闭或转移
4. 两地时间流速或通信延迟不同

### 6. 信息与债

1. 一方知道秘密但不知道秘密的价值
2. 旧债的真实债权人另有其人
3. 消息经由不可靠渠道传播
4. 一方以为交易已经完成，另一方认为只是开始
5. 证词、录像或记录存在可疑缺口

### 7. 信任与验证

1. 一方愿意合作，但无法确认另一方是否诚实
2. 关键帮助必须先通过一次可验证的试探
3. 信任上升会开放更多信息，同时增加被背叛的代价
4. 双方都在等待对方先承担风险

### 8. 责任与义务

1. 个人愿望与身份职责发生冲突
2. 帮助一方会让另一方违反对组织的承诺
3. 角色必须在公开责任和私人选择之间取舍
4. 失职会牵连无关人员或担保人

### 9. 声誉与曝光

1. 私下选择可能损害公开身份
2. 关系越接近，越容易被第三方发现
3. 一份记录或照片足以改变公众看法
4. 角色必须决定保留秘密还是主动公开

### 10. 依赖与撤离

1. 一方需要另一方，但又想恢复独立
2. 帮助越多，关系越难回到原状
3. 退出会失去资源、保护或情报
4. 继续依赖会产生新的责任和情感风险

### 11. 交换与代价

1. 获得关键资源必须付出另一种资源
2. 交换条件本身可能暴露真实关系
3. 一方认为交易已经完成，另一方认为只是开始
4. 替代方案更安全，但会让目标变得更昂贵

### 12. 目标错位

1. 双方都想解决危机，但最终目的不同
2. 一方想保护关系，另一方想利用关系
3. 共同完成短期目标会损害长期目标
4. 任何一方都无法确认对方的最终计划

### 13. 归属与边界

1. 双方争夺同一空间、资源或决定权
2. 公共权限与私人权限范围不一致
3. 一方把照顾理解为保护，另一方理解为控制
4. 必须重新定义谁可以进入、使用或决定某件事

### 14. 记忆与解释

1. 同一行为被双方理解成不同含义
2. 一方记得承诺，另一方只记得试探
3. 过去的许可或拒绝被错误延伸到现在
4. 新证据可能迫使角色重写对过去的判断

### 15. 照护与控制

1. 保护对方会逐渐限制对方的选择
2. 被照护者必须决定接受帮助还是保留自主
3. 照护者的动机同时包含责任和私人愿望
4. 一旦照护关系公开，双方都要承担社会后果

### 16. 规则与例外

1. 制度要求一致处理，但某人获得特殊待遇
2. 例外能解决当前问题，却会破坏规则的可信度
3. 角色必须决定公开例外还是隐瞒来源
4. 例外被撤销后，依赖它的人会立即失去保护

### 17. 距离与沟通

1. 双方想表达真实意图，却只能通过第三方传递
2. 时间差或通信延迟让回应失去原本的含义
3. 面对面更危险，远程沟通又容易误解
4. 必须决定等待确认，还是根据不完整信息行动

### 18. 选择与不可逆

1. 当前决定会关闭未来的一条路径
2. 公开身份、关系或证据后无法完全撤回
3. 不选择也会被解释为一种选择
4. 角色必须在安全方案和真实愿望之间取舍

### 咬合

双引擎必须形成因果连接：

```text
因为 <主引擎压力>，所以 <对冲引擎> 变得更昂贵或更危险。
```

禁止把两个标签简单并列。

| 竞赛与展示 | 选拔现场、行业峰会、评审厅、颁奖晚宴 |
| 危机与避难 | 临时安置点、隔离区、补给站、救援指挥所 |

正式结果固定为：

```yaml
setting_shell:
  type: null
  place: null
  rule: null
  pressure: null
```

```text
<场合类型>中的<具体地点>；
当前规则是<限制>，它让<人物/玩家>必须在<压力>下做选择。
```

`rule` 只写当前最有效的一条限制，例如“非董事不得接触原始文件”或“十点后客区封闭”。`pressure` 必须说明它如何让至少一个开场选择更难或更贵；无关剧情的监控、权限、出口或条款不生成。

壳只提供制度和场合，不自动决定角色态度、同意或关系结果。封闭环境、身份差或时间紧迫也不能替代当前许可。

| 普通人入口 | 新住户、临时雇员、目击者、返乡者、意外继承人、短期租客、自由职业者、前员工 |

身份生成后只需补齐以下最小关系：

```yaml
identity_profile:
  public_identity: null
  actual_function: null
  authority_source: null
  key_resource: null
  limitation: null
  obligation: null
  social_risk: null
  hidden_mismatch: null
  public_reputation: null
  private_context: null
  exposure_risk: null
```

身份必须能解释资源、限制和社会风险；`hidden_mismatch` 可为空，不要求每名角色都有隐藏身份。身份侧描述“这个人凭什么能做什么”，不替代处境侧的当前压力，也不替代角色的态度或同意。

涉及反差方向时，优先生成公开身份与私下场景的差异，再由独立的亲密画像决定欲望、偏向和表现。身份本身不等于性感、淫荡或同意；`exposure_risk` 只表示身份暴露的社会代价。

## 处境侧

处境决定开局时角色正在承受什么，不决定其服从或同意。

| 类型 | 处境种子 |
|---|---|
| 外部逼迫 | 被组织催促签字、公开表态或交出证据 |
| 能力失控 | 能力不稳定，需要合作但不必无条件接受 |
| 身份锁 | 隐藏身份即将暴露，必须选择谁先知道 |
| 误入环境 | 被困在错误地点、卷入错误会议或被误认 |
| 自投翻车 | 主动试探后承担了超出预期的责任 |
| 主动对等 | 双方为共同目标临时结盟，各自保留退出权 |
| 名节舆论 | 流言、偷拍视频或公开误读即将扩散 |
| 资源死线 | 关键资源将在短时间后失效或转移 |
| 主场考校 | 一方在熟悉领域检验另一方的能力和诚意 |
| 公开与私下 | 必须在众人面前维持体面，只有短暂独处窗口 |
| 角色反转 | 原本掌握资源的一方突然需要另一方帮助 |
| 秘密误解 | 一方知道隐情，另一方误以为对方已经做出选择 |
| 共享空间 | 双方必须共处，但仍各自保留暂停和退出权 |
| 关系暴露 | 私人互动可能被熟人、同事或组织发现 |
| 事后余波 | 一次试探或越界后，双方仍必须继续共事 |
| 经验落差 | 一方熟悉这类关系，另一方正在判断自己是否想继续 |
| 资源依赖 | 只有对方能解决当前问题，但交换会暴露真实关系 |

处境正式结果只需补齐以下字段：

```yaml
situation:
  type: null
  trigger: null
  pressure: null
  immediate_objective: null
  deadline: null
  unresolved_choice: null
  knowledge_gap:
    player_knows: []
    npc_knows: []
    both_mistake: []
  exits:
    available: true
    cost: null
    blocked_by: null
  consequence:
    immediate: null
    near_term: null
```

`type` 说明处境形态，`trigger` 说明刚刚发生的事，`pressure` 说明当前压力，`immediate_objective` 说明角色眼下想解决什么，`deadline` 和 `unresolved_choice` 共同制造行动窗口。`knowledge_gap` 只描述信息差，不替角色补写意志。

`exits` 必须明确角色能否暂停、离开或拒绝；封闭环境、身份差、时间紧迫和资源依赖不能自动变成同意。`consequence` 只写不处理的剧情后果，不把压力直接写成角色愿望。

生成局势句：

```text
<角色> 因为 <触发事件>，正在 <地点/关系> 中面对 <当前压力>；
必须在 <期限> 前决定 <未完成选择>，否则 <可见后果> 将发生。
```

## 名字生成与去重

名字由生活背景决定，不由性格决定。先建立命名背景，再生成候选：

```yaml
name_profile:
  culture: null
  region: null
  era: null
  social_context: null
  usage: legal_name
```

中文姓名先按背景分别抽取姓与名，再组合至少 5 个候选。按固定规则评分，禁止按“最有故事感”选择：文化与地区 30、年代 20、自然常见度 20、同局可区分 15、近期不重复 15；文学象征过强扣 20。主 NPC 只需自然、好记、可区分，配角应更普通。

候选必须检查同局重名、读音冲突、近期姓名/姓氏/读音重复和文化一致性。若存在 `references/name_profiles.yaml`，优先读取其中的权重与避用组合；不存在时直接按本节规则生成，不阻塞开局。

姓名字段区分：

```yaml
legal_name: null
public_name: null
nickname: null
alias: null
```

只填写剧情实际需要的字段，不为每个角色强行创建昵称或化名。最近使用记录只参与去重，不得反向改变玩家预锁姓名。

## 人物生成原则

先生成普通人格、目标、底线和生活处境，再独立生成亲密画像。亲密画像不能覆盖人物原本的职业、道德、关系目标或日常行为，也不能从普通人格直接推导。

禁止以下绑定：

```text
善良 ≠ 保守
单纯 ≠ 欲望低
经验少 ≠ 表现被动
表现大胆 ≠ 对所有人开放
欲望强 ≠ 自控差
低欲望 ≠ 等待被改变
亲密偏向 ≠ 当前许可
```

生成时先抽取宽泛倾向，再分别生成各字段。倾向只调整概率，不锁定结果；主 NPC 可提高少见组合概率，普通配角以自然普通为主。同局人物不得全部使用同一倾向。

```yaml
profile_weights:
  ordinary_natural: 35
  reserved_sensitive: 15
  inexperienced: 12
  experienced_restrained: 10
  open_active: 8
  playful: 8
  conflicted: 7
  low_desire: 5
```

所有画像仅用于明确为成年人的虚构角色。知识少、经验少、单纯或不懂世事不得被解释为年龄不明或未成年。

## 亲密画像与变化

将亲密画像保存为相互独立的五层状态；不相关场景不必调用这些字段。

```yaml
sexuality_profile:
  background:
    knowledge: low|medium|high
    experience: none|limited|normal|rich
    mistaken_beliefs: []
  drive:
    intensity: low|medium|high
    frequency: low|medium|high
    importance: peripheral|normal|central
    awareness: unaware|uncertain|clear
  preferences:
    initiative: follow|responsive|lead|switch
    pace: gradual|adaptive|direct
    style: tender|natural|playful|intense|experimental|mixed
    emotional_link: separate|optional|preferred|required
    exclusivity: exclusive|selective|open|undecided
    novelty: cautious|curious|seeking
    turn_ons: []
    turn_offs: []
  expression:
    directness: reserved|natural|direct|uninhibited
    teasing: low|medium|high
    verbal_expression: quiet|responsive|expressive|bold
    physical_initiative: low|medium|high
    public_mask: null
    private_style: null
  regulation:
    self_control: stable|variable|poor
    inhibition: low|medium|high
    conflict_or_shame: none|mild|strong
    compulsivity: none|risk|present
    aftermath_need: distance|normal|reassurance|closeness
```

`drive` 表示欲望强度，`preferences` 表示偏向，`expression` 表示外在表现，`regulation` 表示管理欲望的能力。高欲望、高频率或大胆表现不等于 `compulsivity`；只有持续失控并产生负面后果时才记录强迫风险。

### 兴趣来源

角色表现为低兴趣时，先确定来源，不得默认其等待被改变：

```yaml
interest_origin:
  type: stable|contextual|unexplored|defensive|target_specific
  reason: null
```

`stable` 可以长期不变；`contextual`、`unexplored`、`defensive` 或 `target_specific` 也只能依据角色自身选择和实际事件发展，不预设改变结果。

### 变化状态

```yaml
sexuality_development:
  baseline: {}
  current: {}
  plasticity:
    drive: low|medium|high
    preferences: low|medium|high
    expression: low|medium|high
    regulation: low|medium|high
  change_conditions:
    - sustained_trust
    - voluntary_exploration
    - repeated_positive_experience
    - self_acknowledged_desire
  resistance:
    - pressure
    - dependency_exploitation
    - boundary_violation
  evidence: []
  trend: stable|opening|intensifying|conflicted|retreating
```

欲望、偏向、表现和调节分别变化，不得整卡翻转。至少积累两条跨回合且方向一致的事件证据，并满足对应可塑性与变化条件，才更新 `current`；保留基线、证据和回退可能。

角色可以保持不变、学会表达、发现未知偏好、主动追求更强体验，或尝试后进一步确认拒绝。所谓“沉沦”只可作为正文中的主观叙事效果，后台必须记录为具体字段变化，不能自动扩大许可。

任何调教或亲密探索都必须由成年人自愿进入、能够随时暂停或退出。权力压力、依赖利用、惩罚、醉酒、边界违反或无法拒绝的处境不能成为有效变化证据。

    trust_up: null

每回合按以下顺序决策：

1. 检查年龄、边界、当前许可和安全状态。
2. 只读取该角色的 `knowledge.known`、误信与本回合新获知内容。
3. 选择当前最紧迫目标，并按压力等级确定行动方式。
4. 仅在相关场景读取亲密画像；由偏向决定想要什么，由表现决定如何表达，不得据此推导许可。
5. 排除违反硬底线、玩家控制权或自主冷却的行动。
6. 选出一个行动，内部记录原因；原子提交结果、记忆和关系变化。
7. 记录变化证据；仅在证据、可塑性和条件同时满足时更新当前画像。

亲密画像只描述欲望、偏向和表达方式，不参与许可判定。许可始终只来自独立的当前场景同意记录。
